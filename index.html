Roadmap técnico

O que vamos construir — em 3 horizontes

Curto prazo  ·  1–3 meses

  • Fundação de processo — Definição de Pronto, critério de aceite e branch protection com revisão obrigatória em todos os repos.
  • CI como portão de qualidade — todo PR dispara lint, typecheck e testes. Código que quebra não chega a staging.
  • Sistema de backup diário — snapshots automáticos de Postgres em todos os produtos, com restore drill mensal documentado.
  • Secrets sob controle — secret manager por ambiente, rotação periódica, zero credencial em git.
  • Correções críticas — rotacionar credenciais expostas, limpar o histórico git de segredos e node_modules, restringir CORS por whitelist em todos os produtos e migrar uploads do orka-home para R2.

Médio prazo  ·  3–6 meses

  • Cobertura de testes — meta: 100% no backend, 50% no frontend.
  • Ambientes isolados — staging e produção claramente separados para os produtos principais, com promoção por política definida e rollback em um comando.
  • Infra como código — Railway + R2 documentados e reproduzíveis. A arquitetura deixa de existir só na cabeça das pessoas.
  • Observabilidade real — error tracking, alertas de latência e uptime monitoring ativos. Incidentes descobertos pelo time, não pelo cliente.
  • Dados protegidos por design — PII minimizado, acessos auditados. Processo de compliance LGPD.

Longo prazo  ·  6–12 meses

  • Disaster Recovery formalizado — RTO < 4h e RPO < 1h definidos por produto. Runbook validado em drills trimestrais — não exercício de papel.
  • Redundância em pontos críticos — failover automático para banco e cache nos produtos com receita; backups em região distinta da primária.
  • SLOs — disponibilidade e latência monitoradas com metas explícitas.
  • Segurança — OWASP (referência das vulnerabilidades), rate limiting e Firewall.
  • Pronto para escala — capacity planning baseado em métricas reais.

Design System ORKA

Cada produto do grupo foi construído de forma independente — sem tokens de design, componentes, paleta ou tipografia compartilhados. O resultado é inconsistência visual entre produtos da mesma marca.

O que é o Design System ORKA

  • Biblioteca de componentes React compartilhada
  • Tokens de design: cores, tipografia, espaçamento, sombras
  • Diretrizes visuais e de UX para toda a família ORKA
  • Publicado como pacote interno (npm privado ou monorepo)

Base sugerida

  • shadcn/ui + Tailwind CSS (já em uso em orka-closer e orka-pages)
  • Radix UI primitives para acessibilidade
  • Storybook para documentação viva dos componentes

Benefícios esperados

  • Identidade visual consistente em todos os produtos
  • Velocidade de desenvolvimento frontend: componentes prontos e validados
  • Redução de retrabalho de UI entre produtos

Produtos que se beneficiam imediatamente

  • orka-home (Next.js — sem sistema de design definido)
  • orka-save (painel principal + outreach panel desconexos)
  • Leads-Orka (frontend React sem padronização)

Cultura de engenharia · AI-driven

Anti Vibe Coding Pattern

Devemos guiar a IA para evitar alucinações e códigos indesejados.
1
PRD em Markdown

Toda feature começa com um Product Requirements Document — problema, escopo, critério de aceite, tradeoffs. Versionado ao lado do código que descreve.

2
IA escreve os testes (TDD)

Antes de qualquer linha de implementação, a IA traduz o PRD em testes. Os testes descrevem o comportamento esperado e viram o contrato objetivo da feature.

3
Implementação guiada pelos testes

A IA escreve o mínimo de código necessário para os testes passarem.

4
Iteração até verde

Testes rodam, falhas são analisadas, código é refinado em ciclos curtos. A feature está pronta quando está robusta — não apenas "funcionando".

5
PRD evolui com o código

Ao fim do ciclo, decisões de implementação, tradeoffs descobertos e mudanças de escopo voltam ao documento. PRD nunca fica desatualizado — vive com o código.

Stack aprovada

  • Claude Code — agente principal de implementação
  • Cursor — IDE com edição contextual

Política de dados

  • Sem PII, segredos ou credenciais em prompts
  • Auth, billing e integrações sensíveis revisadas por humano antes do prompt
  • Logs de uso de IA auditáveis por projeto

Métricas que acompanhamos

  • Tempo de feature: do PRD até produção
  • Bugs em produção pós-merge — sinal de TDD fraco
  • Tempo de code review humano
Proposta de sociedade

Termos da proposta

Participação societária

  • 33% da empresa
  • Vesting de 12 meses: início em 20%, atinge 33% ao final do período
  • Vesting condicionado ao cumprimento das metas de curto (1–3 meses), médio (3–6 meses) e longo prazo (6–12 meses) definidas no roadmap técnico

Papel — CTO

  • Autonomia plena sobre decisões técnicas
  • Liderança Técnica dos produtos do grupo
  • Arquitetura, contratação técnica e roadmap de engenharia
  • Ponto de contato técnico com sócios e investidores

Reinvestimento de pró-labore

  • 0–6 meses: 100% da minha distribuição de lucros reinvestida em desenvolvimento tecnológico
  • A partir de 6 meses: revisão conjunta — continuar reinvestimento por +6 meses ou iniciar retirada
  • Skin in the game direto, além do equity

Garantias e regras

  • Não impacta a distribuição de lucros dos demais sócios
  • Valor reinvestido permanece em posição separada de investimento — fora do caixa operacional, não compõe saldo para retiradas futuras
  • Decisões técnicas finais sob escopo do CTO
